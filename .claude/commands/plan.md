아래 작업을 **절대 바로 시작하지 말고**, 먼저 계획서부터 작성해줘.

작업: $ARGUMENTS

## 1단계 — 계획서 작성
- 먼저 관련 파일을 읽어 현재 상태를 정확히 파악한다. (추측하지 말 것)
- `docs/plans/` 폴더에 `YYYY-MM-DD-<주제>.md` 파일로 계획서를 저장한다. 내용:
  - **목표**: 무엇을, 왜 하는가
  - **바꿀 파일**: 어떤 파일의 어느 부분을 어떻게
  - **방법**: 구체적인 단계 (1, 2, 3...)
  - **리스크·주의**: 깨질 수 있는 것. 특히 petfocus라면 데이터 규칙(records 재계산·soft delete·스키마 버전·개인정보 미추가) 위반 여부
  - **검증 방법**: 브라우저 핵심 흐름, 필요 시 원가 검산(231,000원 / 개당 4,620원)
- 계획서의 핵심을 채팅에 알기 쉽게 요약해서 보여준다.

## 2단계 — 멈추고 승인 대기
- 계획을 보여준 뒤 **반드시 멈춘다.** 내가 "좋아 / 진행해 / OK" 라고 하기 전에는 절대 코드를 수정하지 않는다.
- 내가 고칠 점을 말하면 계획서를 수정한 뒤 다시 승인을 기다린다.

## 3단계 — 승인 후 실행
- 승인받으면 그때 계획서대로 작업한다.
- 작업이 끝나면 deploy-check 기준으로 점검한다. (문제 있으면 배포하지 않고 보고)

작업 내용($ARGUMENTS)이 비어 있으면, 먼저 무슨 작업을 할지 물어본다.
