# skytj-tool

핸드폰/PC 브라우저에서 바로 쓰는 사내 관리 도구 모음. 서버 없이 정적 HTML로만 구성되며 GitHub Pages로 배포한다.

## 작업 방식 (중요)

- **코드를 고치기 전에 계획서부터 쓴다.** 도구(HTML)의 동작을 바꾸는 작업은 바로 시작하지 말고, 먼저 관련 파일을 파악한 뒤 `docs/plans/`에 계획서(목표·바꿀 파일·방법·리스크·검증)를 작성하고 핵심을 요약해 보여준다. 그 다음 **사용자 승인을 받고 나서** 실행한다. (`/plan` 명령어와 동일한 절차 — 사용자가 명령어를 치지 않아도 기본 적용)
- 승인 전에는 코드를 수정하지 않는다. 사용자가 방향을 바로잡으면 계획서를 고친 뒤 다시 승인을 기다린다.
- 단, 오타·문구·문서 수정 같은 사소한 변경은 계획서를 생략하고 바로 해도 된다. 애매하면 계획서를 쓰는 쪽을 택한다.
- 작업 후에는 아래 "검증" 기준으로 점검하고, 문제가 있으면 배포하지 않고 먼저 보고한다.

## 사업 컨텍스트 (펫포커스)

- 일반과세자 (부가세 10%), 2025-07 개업, 업태: 반려동물용 사료 제조업 + 전자상거래
- 주력: 프리미엄 한우 우피껌 (수제 소량 생산 → 배치별 원가·수율 관리가 핵심)
- 기준 단가(원): 우피 15,000/kg · 식초 2,500/L · 스틱 150/개 · 포장 450/세트 · 시급 10,000 · 건조 전기 500/h — 앱 설정 탭에서 사용자 변경 가능하므로 코드 기본값(`PRICE_DEF`)과 문서를 함께 갱신할 것
- 답변/설계 관점: 숫자·증빙·비용 효율 우선. 관리회계(앱)와 세무 증빙(홈택스 원본)을 혼동하지 않게 문구 작성

## 개발 원칙

- **도구 1개 = HTML 파일 1개.** 빌드 도구·프레임워크·npm 의존성 금지. CDN 스크립트는 xlsx 등 최소한만.
- 모바일 우선 (390px 기준), 한국어 UI, Pretendard 폰트, 기존 도구와 동일한 CSS 변수 팔레트 유지.
- 데이터는 `localStorage` 저장. 펫포커스 앱의 키는 `petfocus_v1`, 스키마 변경 시 키의 버전을 올리고 `load()`에서 마이그레이션.
- 재고는 저장된 스냅샷이 아니라 **기록(records)에서 매번 전량 재계산** (`computeStock()`). 이 원칙을 깨면 동기화 병합이 망가진다.
- 삭제는 soft delete (`deleted: true`) — 두 기기 병합 시 삭제가 전파되어야 하므로 배열에서 제거하지 않는다.
- 동기화는 Firebase RTDB REST(GET → 병합 → PUT), 병합은 record `id` + `updatedAt` 최신 우선. 서버 코드 작성 금지.
- 고객 개인정보(주소·전화번호)를 펫포커스 앱 데이터 모델에 추가하지 않는다 (공개 URL 동기화 전제이므로).

## 검증

자동 테스트 없음. 변경 시 Playwright(`executablePath: '/opt/pw-browsers/chromium'`)로 파일을 직접 열어 핵심 흐름(매입→생산→판매→재고 반영→새로고침 유지)을 확인하고 콘솔 오류를 점검한다. 원가 검산 기준: 우피10kg·식초2L·스틱50·포장50·건조12h·인시4 → 배치 총원가 231,000원, 50개 생산 시 개당 4,620원.
